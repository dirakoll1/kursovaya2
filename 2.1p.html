2.1. Методы HTTP



Чтобы понять, как работают REST API, важно понимать, что такое ресурсы. Ресурсом может быть любая информация, которая может быть названа, например, документ или изображение, коллекция других ресурсов, невиртуальный объект и многое другое. Между тем, REST использует идентификатор ресурса для распознавания конкретного ресурса, участвующего во взаимодействии между компонентами.


RESTful сервисы обеспечивают гибкость при создании приложений, включающих следующие операции (также известные как CRUD):

• Create

• Read/Retrieve

• Update

• Delete


При выполнении этих операций в RESTful сервисах существуют рекомендации или принципы использования HTTP методов. Эти рекомендации помогают нам выбрать правильные HTTP методы для определенного типа операций. Метод – это тип запроса, который вы отправляете на сервер. Четыре основных ресурсных метода, которые ассоциируются с API REST, следующие:

• GET: Этот метод позволяет серверу найти запрошенные вами данные и отправить их вам обратно.

• PUT: Если вы выполните запрос «PUT», то сервер обновит запись в базе данных.

• POST: Этот метод позволяет серверу создать новую запись в базе данных.

• DELETE: Этот метод позволяет серверу удалить запись в базе данных.



1. Метод GET

Метод HTTP GET используется для получения информации из REST API. Мы не должны использовать этот метод для изменения какой-либо информации или ресурса. GET должен быть идемпотентным, то есть независимо от того, сколько раз он повторяется с одними и теми же параметрами, результаты будут одинаковыми. Многократные запросы должны возвращать одни и те же данные для запроса GET до тех пор, пока другие процессы не изменят их (запрос POST или PUT). HTTP GET запрос должен возвращать ответ 200(OK), если он находит данные.


Рассмотрим пример URI GET запроса:

GET http://www.javadevjournal.com/customers (получить всех клиентов)

GET http://www.javadevjournal.com/customers/1 (получить клиента с ID



2. Метод POST

Этот метод в REST API следует использовать только для создания нового ресурса. В некоторых случаях метод POST используется для обновления сущности, но эта операция не очень частая. Вызов POST API не является идемпотентным.


POST может использоваться одновременно для всех действий изменения:

POST /book/ – добавить книгу (данные в теле запроса)

POST /book/3 – изменить книгу (данные в теле запроса)

POST /book/3 – удалить книгу (тело запроса пустое)



3. Метод PUT

Если мы хотим обновить существующий ресурс, для этой операции следует использовать метод PUT. Метод PUT имеет некоторые особенности по сравнению с методом POST. Метод PUT является идемпотентным. Это означает, что клиент может послать один и тот же запрос несколько раз, но он будет иметь точно такой же эффект, как и однократная отправка.

Если новый ресурс создан с помощью запроса PUT, мы должны сообщить об этом клиенту, отправив правильный ответ (201 Created code).


Некоторые примеры метода PUT:

PUT /book/ — добавить книгу (данные в теле запроса)

HTTPS PUT http://www.javadevjournal.com/customers/1/addresses/123 (Изменение адреса на идентификатор 123)



4. Метод DELETE

Метод DELETE должен использоваться для удаления данного ресурса. Подобно методу PUT, операция DELETE является идемпотентной. Если ресурс удален, любой последующий запрос изменит что-либо в системе, и он должен вернуть ответ 404.


Пример URI метода DELETE:

HTTPS DELETE http://www.javadevjournal.com/customers/1



На каждый HTTP-запрос сервер отвечает HTTP-ответом, который всегда содержит код состояния HTTP. В ответе указывается статус запрошенной операции. Чаще всего это 200 (OK). Это способ сервера сказать, что все в порядке.


Наиболее используемыми кодами состояния являются:

• 200 (OK) – Пример: Запрос был выполнен без проблем

• 201 (Created) – Пример: В базе данных создан новый пользователь

• 401 (Unauthorized) – Пример: Пользователь не имеет прав на URL

• 404 (Not Found) – Пример: Документ с указанным URL не существует
